## System Intent and End-to-End Flow

Setlist to Playlist converts a `setlist.fm` URL/ID into an Apple Music playlist using a split
server/client trust model.

1. `apps/web` collects setlist input from the user.
2. `apps/web` API routes delegate server tasks to `apps/api` handlers.
3. `apps/api` proxies setlist requests using `SETLISTFM_API_KEY` (server-side only).
4. `apps/api` mints MusicKit developer JWTs from Apple key env vars.
5. Browser-side matching/playlist logic uses `packages/core` utilities and MusicKit calls.
6. Shared constants/types/errors are centralized in `packages/shared`.

## Area-by-Area Audit

### 1) `apps/web` (UI + API routes)

**What works well**

- Feature-based organization (`setlist-import`, `matching`, `playlist-export`) is clear.
- API routes include CORS handling and route-level abuse controls.
- Tests cover hooks, route behavior, and matching/playlist helper logic.

**Findings and actions**

- **P1 remediated:** when `TRUST_PROXY` was unset, rate-limit identity could collapse to one
  shared key (`unknown`), allowing one caller to throttle all users.
  - **Fix:** derive anonymous fallback keys from non-IP headers (`user-agent`,
    `accept-language`, `origin`, `host`) instead of using one global key.
  - **Added tests:** verify derived fallback and explicit fallback when no fingerprint exists.

### 2) `apps/api` (serverless logic)

**What works well**

- Dev-token handler validates required env vars and avoids private-key logging.
- setlist fetcher has response size limits, retry-after parsing, and 429 retry strategy.

**Findings and actions**

- **P2 remediated:** setlist proxy previously surfaced upstream error message text directly to
  clients (length-truncated, but still noisy/unstable).
  - **Fix:** normalize client-facing messages by status class (404/429/5xx/other).
  - **Added tests:** verify 429 stable message and 400 normalization behavior.

### 3) `packages/core` (domain logic)

**What works well**

- Good test depth for parsing/normalization/setlist mapping edge cases.
- Clear pure-function boundaries suitable for reuse by UI and API.

**Opportunities (P3 backlog)**

- Consider exposing a few composite helpers for repeated app-level orchestration paths
  (import→map→dedupe) to reduce glue code drift over time.

### 4) `packages/shared` (shared contracts/utils)

**What works well**

- Shared API error constants and message types reduce drift across apps.
- Utility functions are narrow and test-backed.

**Opportunities (P3 backlog)**

- Consolidate user-facing message catalog coverage (some route-specific strings still local).

### 5) `packages/ui`

**What works well**

- Lightweight placeholder package with basic component test coverage.

**Opportunities (P3 backlog)**

- Define promotion criteria for component migration from `apps/web/src/components` into
  `packages/ui` to avoid future duplication.

### 6) Scripts and operational tooling

Reviewed scripts:

- `scripts/export-diagnostics.ts`
- `scripts/cleanup-repo.sh`
- `scripts/seed-demo-setlists.ts`

**What works well**

- Utilities are purposeful and scoped to support/dev workflows.

**Opportunities (P3 backlog)**

- Add a script-level test harness or smoke checks to guard behavioral regressions for non-trivial
  script logic.

## Cross-Cutting Refactor / Dedup / Optimization Opportunities

1. **Distributed rate limits/cache (P3):** current in-memory stores are per-instance; consider
   optional shared adapters for scaled deployments.
2. **String/message centralization (P3):** unify remaining route-local message constants into
   shared message modules where beneficial.
3. **Observability (P3):** add structured counters around upstream failures, retries, and route
   throttling to make production tuning easier.
4. **Build/lint environment consistency (P3):** ensure lint dependency resolution is reproducible
   in all CI/dev environments.

## Remediation Implemented in This Iteration

### P1: Rate-limit fallback identity hardening

- File: `apps/web/src/lib/rate-limit.ts`
- Change: replaced single shared fallback behavior with derived anonymous keying when proxy
  headers are untrusted.
- Benefit: avoids accidental global throttling while preserving anti-abuse behavior.

### P2: Stable/safe proxy error messages

- File: `apps/api/src/routes/setlist/proxy.ts`
- Change: replaced upstream message passthrough with deterministic client-safe message mapping.
- Benefit: improved UX consistency and reduced upstream detail leakage.
